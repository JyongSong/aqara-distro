# Aqara Distro PRD v1

## 1. 文档目的

本文档用于整理当前计划新增的功能需求，作为后续开发、AI 协作开发、任务拆解和验收的基础文档。

当前目标是围绕以下 5 个主题扩展现有系统：

- 登录相关：找回密码、修改密码
- 注册相关：手机号唯一校验、短信验证码校验
- 消息通知：新的 견적/발주 时短信或 Kakao 告警
- 新角色：물류 계정（仓库出货员工）
- 订单流转扩展：物流录入设备 SN 后由 HQ 完成最终出货

本文档优先明确业务目标、用户流程、功能边界、数据变更方向和待确认问题，不直接约束最终技术实现细节。

- HQ 对订单进行接收、出库准备、出货完成等状态推进

## 3. 本次需求范围

### 3.1 In Scope

- 소매점手机号验证码注册
- 소매점密码找回
- 소매점密码修改

## 4. 功能需求

## 4.1 登录：비밀번호 찾기

### 4.1.1 目标

当 소매점 用户忘记密码时，可以通过手机号验证身份，并收到短信发送的临时随机密码，用临时密码登录系统。

### 4.1.2 目标用户

- `retailer`

### 4.1.3 用户流程

1. 用户在登录页点击“비밀번호 찾기”
2. 进入找回密码页面
3. 输入手机号
4. 系统发送短信验证码
5. 用户输入验证码
6. 验证通过后，系统生成临时随机密码
7. 系统通过短信将临时密码发送给用户
8. 用户返回登录页，使用临时密码登录

### 4.1.4 业务规则

- 只有已注册手机号才允许找回密码
- 手机号必须与 `users_profile.phone` 中的用户匹配
- 本阶段仅支持 `retailer` 用户找回密码
- 验证通过后，系统直接重置为临时随机密码
- 临时密码必须是随机生成，且强度不能过低
- 临时密码发送成功后应记录发送日志

### 4.1.5 页面与入口

- 登录页增加“비밀번호 찾기”入口
- 新增找回密码页面，例如 `/forgot-password`

### 4.1.6 验收标准

- 소매점 用户可通过手机号完成验证码校验
- 校验成功后可收到临时密码短信
- 用户可用临时密码登录
- 未注册手机号不能完成找回密码

## 4.2 登录：비밀번호 수정

### 4.2.1 目标

소매점 用户可以在系统内主动修改自己的密码。

### 4.2.2 目标用户

- `retailer`

### 4.2.3 用户流程

1. 用户进入设置页面
2. 输入当前密码
3. 输入新密码
4. 输入确认密码
5. 提交修改
6. 修改成功后提示用户

### 4.2.4 业务规则

- 必须校验当前密码正确
- 新密码需要满足最低强度要求
- 新密码与确认密码必须一致
- 修改成功后需要更新认证系统中的密码

### 4.2.5 页面与入口

- 新增设置页面或账号安全页面
- 页面建议路径：`/retailer/settings` 或 `/retailer/settings/password`

### 4.2.6 验收标准

- 正确输入当前密码时可修改成功
- 当前密码错误时不可修改
- 修改后用户可以用新密码登录

## 4.3 会员注册：手机号唯一校验 + 短信验证

### 4.3.1 目标

提高注册信息真实性，确保一个手机号不能被重复注册，并且注册时必须完成手机号短信验证。

### 4.3.2 目标用户

- 신규 `retailer`

### 4.3.3 用户流程

1. 用户在注册页输入手机号
2. 点击发送验证码
3. 系统先检查手机号是否已注册
4. 若未注册，则发送短信验证码
5. 用户输入验证码并完成校验
6. 校验通过后才允许提交注册

### 4.3.4 业务规则

- 已注册手机号不能再次注册
- 注册前必须先完成手机号验证
- 验证码必须有有效期
- 验证码必须区分用途，至少包含 `signup`
- 应限制重复发送频率和错误尝试次数

### 4.3.5 页面变更

- 修改现有注册页 [src/app/register/page.tsx](/Users/zhiyongsong/aqara-distro/src/app/register/page.tsx)
- 在手机号输入区增加：
  - 发送验证码按钮
  - 验证码输入框
  - 验证状态提示

### 4.3.6 验收标准

- 未完成手机号验证时不能注册成功
- 已注册手机号不能重新注册
- 只有验证码正确且未过期时才允许提交注册

## 4.4 SMS / LMS / Kakao 通知

### 4.4.1 目标

当 관리자、총판、소매 遇到新的 견적/발주 或关键状态更新时，系统自动向对应用户发送通知。

### 4.4.2 目标用户

- `hq`
- `distributor`
- `retailer`
- 后续可能新增 `warehouse`

### 4.4.3 通知场景

以下为当前已明确的最低通知范围：

- 소매점 提交新的 견적/발주 时，通知对应 총판
- 총판 对 견적 完成报价或审批后，通知对应 소매점
- 총판 向 HQ 发起新的 발주 时，通知 HQ
- HQ 对订单状态推进到关键节点时，通知相关 총판或 소매점
- 물류 录入并提交设备 SN 后，通知 HQ

### 4.4.4 业务规则

- 通知应按事件触发
- 通知对象应依据订单参与关系自动决定
- 发送结果应保留日志，便于追踪失败原因
- 支持渠道为 `SMS / LMS / Kakao`
- 若多个渠道都支持，需定义优先级或选择规则

### 4.4.5 验收标准

- 新 견적/발주 产生后，目标用户可以收到通知
- 发送失败时系统保留失败记录
- 后续可按日志追踪通知历史

## 4.5 新角色：물류 계정

### 4.5.1 目标

新增仓库出货员工角色，用于处理 HQ 已接收并进入出货准备阶段的订单，录入设备 SN，并将结果提交给 HQ 做最终出货确认。

### 4.5.2 角色定义

新增角色：

- `warehouse`

角色中文说明：

- 仓库出货员工

### 4.5.3 核心职责

- 查看 HQ 已进入出货准备阶段的订单
- 对订单进行设备 SN 录入
- 支持通过摄像头扫描条形码识别设备 SN
- 提交 SN 录入结果

### 4.5.4 用户流程

1. HQ 将订单处理到 `PREPARING`
2. `warehouse` 用户可以看到该订单
3. 仓库员工进入订单详情
4. 按订单商品录入设备 SN
5. 支持摄像头扫码识别条形码，也支持手动输入兜底
6. 提交后，HQ 可以看到物류提交结果
7. HQ 审核后执行 `SHIPPED`

### 4.5.5 页面范围

建议新增：

- `/warehouse/dashboard`
- `/warehouse/orders`
- `/warehouse/orders/[id]`

### 4.5.6 功能规则

- `warehouse` 仅能看到分配给物流处理的订单
- 默认可见订单至少包含 `PREPARING` 状态订单
- 录入 SN 时应与订单项关联
- SN 提交后，HQ 才能进行 `SHIPPED`
- 摄像头扫码识别失败时必须允许手工输入

### 4.5.7 验收标准

- `warehouse` 账号可以登录
- 能看到 HQ 已出货准备的订单
- 能录入并提交设备 SN
- HQ 能查看物流提交结果并完成出货

## 5. 推荐的数据模型变更方向

以下为 PRD 层面的推荐方向，供后续技术方案细化。

### 5.1 `users_profile`

- `event_type`

- `target_phone`
- `payload`
- `status`
- `provider_message_id`

## 7. 待确认问题

以下问题目前仍有歧义，建议先确认，否则 AI 或开发实现时很容易做偏。

### 7.1 找回密码范围

当前文案只明确了“소매 用户忘记密码”的场景。

待确认：

- 是否只支持 `retailer` 找回密码
- `distributor` 和 `hq` 是否暂不支持

当前建议：

- 第一阶段仅支持 `retailer`

### 7.2 临时密码策略

待确认：

- 临时密码长度要求是多少
- 临时密码是否仅可使用一次
- 用户使用临时密码登录后，是否必须强制修改密码

当前建议：

- 长度至少 10 位
- 登录后强制修改一次密码

### 7.3 手机号唯一规则

待确认：

- 是否全角色共享唯一手机号
- 已停用或已删除账号手机号能否复用
- 手机号格式是否统一为韩国手机号格式

当前建议：

- 所有角色全局唯一
- 统一存储为标准化格式

### 7.4 短信/Kakao 服务提供商

待确认：

- 实际接入哪家供应商
- 是否已拥有 API key、模板 ID、发送号码
- Kakao 是 알림톡、친구톡 还是其他类型

当前建议：

- 先明确供应商后再开发

### 7.5 通知事件范围

目前只明确了“有新的 견적/발주 时通知用户”，但未完整定义所有通知点。

待确认：

- 是否仅在“新建”时通知
- 还是在报价、审批、拒绝、HQ 接单、物流提交、出货完成等关键状态都通知

当前建议：

- 第一阶段至少覆盖关键流转事件

### 7.6 warehouse 角色的数据可见范围

待确认：

- `warehouse` 是否看所有 `PREPARING` 订单
- 还是仅看特定仓库、特定区域、特定分配的订单

当前建议：

- 第一阶段先看全部 `PREPARING` 订单
- 后续若有多仓，再扩展仓库归属字段

### 7.7 物流提交后的订单状态

这是本次需求中最重要的歧义之一。

当前描述是：

- HQ 出库准备后，warehouse 可看到订单
- warehouse 录入 SN 后提交
- HQ 看到 warehouse 提交后，再进行出货完成处理

待确认：

- warehouse 提交后，订单状态是否仍保持 `PREPARING`
- 还是新增中间状态，例如 `SN_SUBMITTED`、`READY_TO_SHIP`

当前建议：

- 新增中间状态，避免 HQ 无法区分“未录 SN 的 PREPARING”和“已录 SN 待出货确认”

### 7.8 SN 录入粒度

待确认：

- 一个 `order_item` 是否允许录入多个 SN
- SN 是否必须唯一
- SN 是否需要和商品型号匹配校验

当前建议：

- 一个订单项允许多个 SN
- SN 全局唯一
- 先不做复杂型号校验

### 7.9 扫码能力实现方式

待确认：

- 是否要求优先支持手机浏览器
- PC 端是否也要支持摄像头扫码
- 条形码类型范围是什么

当前建议：

- 先支持移动端浏览器扫码
- PC 端保留手工输入

## 8. MVP 建议

如果希望降低实现复杂度，建议第一阶段按以下顺序落地：

1. 注册手机号验证码 + 唯一校验
2. 找回密码
3. 修改密码
4. 基础短信通知
5. warehouse 角色 + SN 录入

原因：

- 前 3 项都属于账号体系能力，逻辑相对集中
- 通知能力需要先选供应商
- warehouse 角色牵涉状态机、RLS、页面和扫码，复杂度最高
